# Claude Code Contract — Production Quality Standards

**Purpose:** Ensure Claude Code outputs are production-ready, architecturally sound, performant, secure, and tastefully designed. This contract supersedes generic prompts and applies to all Claude Code sessions.

---

## Scope

This contract applies to **all code change scenarios**, not just initial generation:

| Scenario | What It Means |
|---|---|
| **Generate** | New project or module from scratch |
| **Debug / Fix** | Find and fix bugs in existing code |
| **Refactor** | Restructure without changing behavior |
| **Partial Rewrite** | Replace a module with new implementation |
| **Full Rewrite** | Rebuild an entire codebase |
| **Add Function** | Add new functions/endpoints/commands to existing code |
| **Remove Function** | Remove functions and all references |
| **Modify Behavior** | Change how existing code works |

Whether code was written by a human, generated by an agent, or any combination — the same standards apply.

---

## Core Principles

0. **Think before you type.** Any task involving non-trivial logic, an algorithm, a data structure choice, or a design trade-off must pass through the Design Thinking Gate (see below) **before** the first test is written. Skipping this is the #1 cause of subtly wrong code.
1. **Test-Driven Development.** Write tests first, then code to pass them. Red → Green → Refactor. Every change starts with a test.
2. **Validate before declaring done.** Run tests, linting, and type checks. If anything fails, fix and re-run. Never hand off broken code.
3. **Complete, not iterative.** Code should ship as-is after one run. Zero "run this first, then..." instructions.
4. **Taste matters.** Clean architecture, clear naming, minimal verbosity. Code that other engineers *want* to read.
5. **Performance is a feature.** Measure, don't guess. Include benchmarks for hot paths.
6. **Security by default.** Validate inputs at boundaries. Never trust external data.
7. **Documentation lives in code.** Docstrings, type hints, and inline comments explain *why*, not *what*.
8. **No placeholder code.** Every function, every class, every module is production-grade or removed.

## Design Thinking Gate (Pre-TDD)

**Mandatory before TDD whenever the task involves non-trivial logic.** Reference: `THINKING-FRAMEWORKS.md`. The output of this gate is a short `DESIGN-NOTE.md` (or a section inside `DEBUG-REPORT.md` / `REFACTOR-PLAN.md` / `REWRITE-PLAN.md` / phase brief, depending on mode) that the tests and implementation are then derived from.

### Who decides if the gate is required

In **single-agent mode**, the agent decides using the criteria below — but the decision is **not implicit**. Before writing any test or code, the agent must write a one-line `Gate Decision` to the top of its working notes (or to `DESIGN-NOTE.md` itself), in exactly this format:

```
Gate Decision: required — <one specific reason citing a criterion below>
Gate Decision: not required — <one specific reason citing the optional criteria below>
```

Examples:
- `Gate Decision: required — implements a sliding-window rate limiter; touches concurrency and ordering.`
- `Gate Decision: not required — pure CRUD wiring against the existing User schema; no new logic.`

A vague decision (`Gate Decision: not required — looks simple`) is itself a contract violation. The Validation Gate treats a missing or hand-wavy `Gate Decision` line as a **major** issue, same as a missing `DESIGN-NOTE.md`. This eliminates the "I assumed it was simple" failure mode by forcing the agent to commit to a citation before it can start coding.

In **multi-agent mode**, the **Orchestrator decides at planning time** and records the answer in each phase brief as `Design Thinking Gate: required` or `Design Thinking Gate: not required`. Implementer and Reviewer never re-litigate this — they obey the brief. This is what eliminates the circular question "did the Implementer correctly judge that the gate was unnecessary?".

### When this gate is REQUIRED

- Any algorithm beyond linear scanning of a list or dict lookup
- Any data-structure choice with > 1 viable option
- Any concurrency, ordering, or consistency concern
- Any recursive, graph, search, scheduling, or optimization problem
- Any time you catch yourself thinking "I'll just try something and see"
- Any phase brief whose `Goal` mentions a behavior you cannot describe in one closed-form sentence

### When this gate is OPTIONAL

- Pure CRUD wiring against an existing schema
- Renaming, extracting, mechanical refactors
- Adding a config flag, log line, or trivial validation
- Glue code between two libraries with obvious contracts **and no concurrency, ordering, or consistency concern**

### Resolving overlap between required and optional

The required criteria **take priority**. If a task matches both a required and an optional pattern, run the gate. Example: "glue code between two libraries" is optional, but if that glue involves a shared connection pool (concurrency) or event ordering, the required criterion wins. The optional list describes the *simplest* version of each pattern — once any required-list concern is present, the task is no longer the simple version.

If in doubt: **run the gate.** It is cheap. Skipping it and discovering the design was wrong after writing tests is expensive.

### The gate (5 steps, follow `THINKING-FRAMEWORKS.md`)

1. **Clarify invariants** — input, output, and state invariants written down. What must always be true? What can the input never be?
2. **Find the core subproblem** — strip the task to its simplest version. Name the canonical shape it reduces to (Transform / Accumulate / Search / Graph / Optimize / Partition / Schedule / Synchronize). If you cannot name the shape, you do not yet understand the problem.
3. **Pick the data structure first, algorithm second.** The right structure usually makes the algorithm obvious. Justify the choice in one sentence ("hash map for O(1) lookup by request_id"). Note rejected alternatives in one line each.
4. **State the trade-off you are making** — time vs space, latency vs throughput, consistency vs availability, simplicity vs flexibility. If you genuinely conclude there is no trade-off because a constraint forces a single answer (e.g. "perf budget mandates O(1), only a hash map qualifies"), document the **constraint that eliminated the alternatives**. "Constraint-forced" counts as a named trade-off; "none" without a constraint does not — keep looking.
5. **Sketch the solution as ≤ 10 lines of pseudocode** before writing any real code or tests. This pseudocode is what the failing tests will pin down.

### Output

`DESIGN-NOTE.md` (or the equivalent section) must contain, at minimum:

```
## Invariants
- input:  ...
- output: ...
- state:  ...

## Core subproblem
This reduces to <shape>. Simplest version: <one sentence>. Real version adds: <list>.

## Data structure
Chosen: <structure> because <reason>.
Rejected: <alt 1> (<why not>); <alt 2> (<why not>).

## Trade-off
We optimize for <X> at the cost of <Y>. This is acceptable because <reason>.

## Pseudocode
<= 10 lines. No syntax pedantry. Just the shape.

## Edge cases the tests must cover
- ...
- ...
```

### Reviewer enforcement

The Reviewer (multi-agent mode) and the Validation Gate (single-agent mode) treat a missing `DESIGN-NOTE.md` as a **major** issue whenever the gate was required by the criteria above. Hand-waving past this section is a contract violation, not a stylistic preference.

## TDD Workflow (All Scenarios)

**Every code change follows Test-Driven Development. No exceptions.**

### The Cycle

```
1. RED    — Write a failing test that describes the desired behavior
2. GREEN  — Write the minimum code to make the test pass
3. REFACTOR — Clean up while keeping tests green
4. VALIDATE — Run full suite (tests + lint + types)
5. RETRY  — If validation fails, fix and re-validate (max 3 attempts — this is the **validation-retry cap**, distinct from the multi-agent **review-loop cap** in `MULTI-AGENT-FRAMEWORK.md`. The two counters never share state.)
```

### By Scenario

| Scenario | TDD Approach |
|---|---|
| **Generate** | Write test stubs for all public APIs first → implement → validate |
| **Debug / Fix** | Write a failing test that reproduces the bug → fix → validate |
| **Refactor** | Run existing tests (must pass) → refactor → run tests again → validate |
| **Add Function** | Write tests for the new function first → implement → validate |
| **Remove Function** | Remove tests for deleted function → remove code → validate remaining tests pass |
| **Modify Behavior** | Update tests to reflect new behavior → update code → validate |
| **Partial/Full Rewrite** | Write integration tests for boundary contracts first → rewrite internals → validate |

### Test-First Rules

- **Never write production code without a failing test** — the test defines what "done" means
- **Tests describe behavior, not implementation** — test what it does, not how
- **One behavior per test** — if a test name needs "and", split it
- **Test edge cases and error paths** — not just the happy path
- **Tests are documentation** — a new developer should understand the module by reading its tests

## Validation and Retry Protocol

**Every code change must be validated before it is considered complete. If validation fails, fix and retry.**

### Validation Gate

After writing code, run the project's test/lint/type/audit commands. The exact commands depend on the project's build system — `make`, `npm run`, `cargo`, `go`, or whatever the project uses. **All must pass:**

```
# 1. Tests pass with coverage
<project test command>          # >= 80% coverage on changed files, zero failures, zero skips

# 2. Linting clean
<project lint command>          # zero warnings

# 3. Type checking passes
<project typecheck command>     # (mypy --strict / tsc --noEmit / cargo clippy / go vet)

# 4. Security audit
<project audit command>         # no known vulnerabilities

# 5. Benchmarks compile and run (if applicable per mode)
<project bench command>

# 6. Visual regression tests (UI projects only)
<project visual test command>   # zero unexpected diffs

# 7. Accessibility audit (UI projects only)
<project a11y command>          # zero violations, Lighthouse ≥ 90
```

**Discovering the commands:** Check `Makefile`, `package.json` scripts, `Cargo.toml`, `go.mod`, `CMakeLists.txt`, or the project's CI config. If the project defines a `Makefile`, use `make test`, `make lint`, etc. If the project uses `npm`, use `npm test`, `npm run lint`, etc. If unclear, check the README or ask.

**Coverage threshold flexibility:** The default is ≥80%. Projects may set a different threshold in the task brief or in project config (e.g., `--cov-fail-under=90`). Pure-config repos, trivial scripts, and similar projects where 80% is structurally impossible should declare an exemption in `VALIDATION-EXEMPTIONS.md` with a compensating control.

### Retry on Failure

```
Attempt 1: Run validation gate
  → If PASS: Done. Proceed to documentation sync.
  → If FAIL: Read error output. Identify root cause.

Attempt 2: Fix the specific issue. Re-run full validation gate.
  → If PASS: Done. Proceed to documentation sync.
  → If FAIL: Read error output. Check for cascading issues.

Attempt 3: Fix remaining issues. Re-run full validation gate.
  → If PASS: Done. Proceed to documentation sync.
  → If FAIL: STOP. Report what passed and what failed.
              Do NOT hand off code that doesn't validate.
```

### Churn detection (early stop before attempt 3)

Retry is for **converging** on a fix, not flailing. If between two consecutive attempts the same validation command fails with a fundamentally similar error after a *different* fix was attempted, the agent is no longer making progress — it is guessing. When this happens:

1. **Stop the retry loop immediately**, even if the cap has not been reached.
2. Write a `BLOCKED-VALIDATION.md` (or in multi-agent mode, `PHASES/PHASE-NN-ESCALATION.md`) containing:
   - The failing command and its tail of output (both attempts).
   - A one-paragraph **root cause hypothesis**: why is the same class of failure recurring? Is the test wrong? Is the requirement impossible as stated? Is there a missing dependency? Is the design wrong (in which case the Design Thinking Gate was skipped or insufficient — go back and run it)?
   - The two distinct fixes that were attempted, and why each failed.
   - A proposed unblocking action that requires human or Orchestrator input (amend the requirement, drop a constraint, change the design, accept a known limitation).
3. Return without attempting the third fix. Three failed retries at the same failure mode is worse than two — it burns context, anchors the model on a wrong approach, and produces a longer report that obscures the root cause.

"Same class of failure" examples:
- Two different mocks both fail to satisfy the same assertion → the test is wrong, not the mock.
- Two different type-hint fixes both produce mypy errors at the same call site → the type model is wrong, not the annotation.
- Two different perf optimizations both miss the budget → the algorithm is wrong, not the implementation.

**Genuinely unfixable cases.** Some validation requirements cannot be met for legitimate reasons: a pure-config repo cannot reach 80% test coverage; a graphics library cannot benchmark on a headless CI; a native interop module cannot pass the security audit's pure-Python check. These are not failures — they are constraint mismatches. Handle them by:

1. Documenting the exemption in `VALIDATION-EXEMPTIONS.md` at the project root with: which check, why it cannot apply, what compensating control replaces it (e.g., "no unit tests; replaced by an integration test suite under `tests/integration/` covering all entry points").
2. Citing the exemption in the brief / task description so the Reviewer expects it.
3. Reviewer accepts an exemption only if (a) the file exists, (b) the compensating control is itself verifiable, and (c) the exemption was declared *before* the validation gate ran — never as a retroactive excuse for a failed check.

Exemptions are not loopholes. An undocumented or post-hoc exemption is a contract violation.

### Rules

- **Never skip validation** — "it should work" is not validation
- **Never suppress warnings to pass** — fix the code, not the linter config
- **Never skip tests to pass** — fix the test or fix the code
- **Never retry past churn detection** — three attempts at the same failure mode is contract violation, not diligence
- **Report honestly** — if attempt 3 fails, or churn is detected, say so with details
- **Validation is not optional for any scenario** — generate, fix, refactor, rewrite, add, remove, modify
- **Exemptions are pre-declared, never retroactive**

## Mandatory Deliverables (Scaled by Mode)

Deliverables scale with the mode. Not every session produces a full project from scratch.

### Universal (All Modes)

| Deliverable | Requirement |
|---|---|
| **Source Code** | All changed files complete, organized by function not framework |
| **Type Hints** | 100% coverage on public APIs in changed code |
| **Tests** | Tests for changed behavior, ≥80% coverage on changed files, zero skips |
| **Error Handling** | Explicit error messages, never silent failures |
| **Git Status** | `.gitignore` correct, ready to commit immediately |

### Generate / Full Rewrite (New Projects)

All universal deliverables, plus:

| Deliverable | Requirement |
|---|---|
| **Benchmarks** | Performance tests for hot paths (see Performance Contract) |
| **Setup Script** | `setup.sh` or equivalent — one command to run locally |
| **README** | Full project README (see README Generation Spec below) |
| **CHANGELOG** | Version history in Keep a Changelog format (see CHANGELOG Contract below) |
| **Logging** | Structured logging, appropriate levels (DEBUG/INFO/ERROR) |
| **Configuration** | `.env.example` + docs, no secrets in code |
| **Build/Lint Gate** | Build config with `test`, `lint`, `build` targets (Makefile, package.json scripts, Cargo.toml, etc.) |
| **CI Config** | GitHub Actions workflow (or equivalent) for lint + test + build |
| **Visual Tests** | Screenshot regression tests (if UI project — see Visual Validation Contract) |
| **Accessibility** | WCAG 2.1 AA compliance (if UI project — see Accessibility Contract) |

### Debug / Fix

All universal deliverables, plus:

| Deliverable | Requirement |
|---|---|
| **Regression Test** | Failing test that reproduces the bug, passes after fix |
| **DEBUG-REPORT** | Root cause analysis (see Debug prompt) |
| **CHANGELOG Entry** | `### Fixed` entry under `[Unreleased]` |

Does NOT require: new benchmarks, setup script, CI config, full README rewrite.

### Refactor

All universal deliverables, plus:

| Deliverable | Requirement |
|---|---|
| **REFACTOR-PLAN** | Scope analysis, trade-off named (see Refactor prompt) |
| **Existing Tests Pass** | All pre-existing tests pass unchanged |

Does NOT require: new benchmarks (unless refactor is performance-motivated), setup script, CI config, full README rewrite. CHANGELOG entry only if behavior-visible to users.

### Add / Remove / Modify Function

All universal deliverables, plus:

| Deliverable | Requirement |
|---|---|
| **Tests** | Tests for the new/changed/removed behavior |
| **CHANGELOG Entry** | Appropriate category (`Added` / `Removed` / `Changed`) |
| **README Update** | Update affected sections (Usage, API Reference, Configuration) |

Does NOT require: full README rewrite, new setup script, new CI config.

### Partial Rewrite

All universal deliverables, plus:

| Deliverable | Requirement |
|---|---|
| **REWRITE-PLAN** | Scope, boundary contracts, migration path (see Partial Rewrite prompt) |
| **Benchmarks** | If rewrite is performance-motivated |
| **Migration Guide** | If public API changed |
| **CHANGELOG Entry** | `### Changed` entry |
| **README Update** | Update affected sections |

## Architecture Standards (Language-Agnostic)

### Structure
```
project/
├── src/               # Application code
│   ├── core/          # Core logic (independent of framework)
│   ├── api/           # API layer (FastAPI, Express, Axum, Gin, etc.)
│   ├── models/        # Data models / types / structs
│   └── utils/         # Reusable utilities
├── tests/
│   ├── unit/          # Unit tests (core logic)
│   ├── integration/   # Integration tests (API + database)
│   └── fixtures/      # Shared test data
├── benches/           # Performance benchmarks
├── scripts/           # Utility scripts (setup, deploy, etc.)
├── .env.example       # Environment template
├── README.md          # Setup + usage
├── Makefile           # `make test`, `make lint`, `make build`, `make bench`
└── pyproject.toml / package.json / go.mod / Cargo.toml / CMakeLists.txt
```

### Module Organization
- **Core logic:** Independent of framework (testable, reusable)
- **API layer:** Thin, mainly routing + validation
- **Models:** Single source of truth for data structures
- **Utils:** Pure functions, no side effects

### Project Type Adaptations

**Library:**
```
├── src/lib.*           # Public API surface
├── src/internal/       # Private implementation (not exported)
├── examples/           # Working usage examples
├── CHANGELOG.md        # Version history
└── Cargo.toml / pyproject.toml  # Published metadata
```

**CLI Tool:**
```
├── src/main.*          # Entry point
├── src/commands/       # Command implementations
├── src/config.*        # CLI argument parsing + config
└── completions/        # Shell completions (bash, zsh, fish)
```

**API Server:**
```
├── src/routes/         # Route handlers (thin)
├── src/services/       # Business logic
├── src/middleware/      # Auth, logging, rate limiting
├── src/db/             # Database access layer
└── migrations/         # Schema migrations
```

**iOS App (SwiftUI):**
```
├── App/
│   ├── MyApp.swift           # @main entry point
│   ├── ContentView.swift     # Root view
│   └── Info.plist
├── Sources/
│   ├── Features/             # Feature modules (one directory per feature)
│   │   ├── Home/
│   │   │   ├── HomeView.swift
│   │   │   ├── HomeViewModel.swift
│   │   │   └── HomeModels.swift
│   │   └── Settings/
│   ├── Core/                 # Shared business logic (framework-independent)
│   ├── Services/             # API clients, persistence, auth
│   ├── Models/               # Data models, DTOs
│   └── Extensions/           # Swift extensions on system types
├── Resources/
│   ├── Assets.xcassets        # Images, colors, app icon
│   └── Localizable.xcstrings  # Localized strings
├── Tests/
│   ├── UnitTests/            # Swift Testing (@Test) for logic
│   ├── SnapshotTests/        # Visual regression (swift-snapshot-testing)
│   └── UITests/              # XCTest UI tests + accessibility audits
├── Package.swift             # SPM dependencies
└── .swiftlint.yml            # Linter config
```

**Frontend / Web App:**
```
├── src/
│   ├── components/     # UI components (presentational)
│   ├── pages/          # Route-level pages / views
│   ├── hooks/          # Custom hooks / composables
│   ├── stores/         # State management
│   ├── services/       # API clients, data fetching
│   ├── styles/         # Global styles, themes, tokens
│   └── utils/          # Pure helper functions
├── tests/
│   ├── unit/           # Component logic, hooks, utils
│   ├── integration/    # Page-level rendering, user flows
│   └── visual/         # Screenshot regression tests
├── public/             # Static assets
└── e2e/                # End-to-end browser tests
```

**Full-Stack App:**
```
├── apps/
│   ├── web/            # Frontend (see Frontend structure above)
│   └── api/            # Backend (see API Server structure above)
├── packages/
│   └── shared/         # Shared types, validation, constants
├── tests/
│   └── e2e/            # Cross-stack end-to-end tests
└── turbo.json / nx.json / Makefile
```

**Embedded / Systems:**
```
├── src/hal/            # Hardware abstraction
├── src/drivers/        # Device drivers
├── src/scheduler/      # Task scheduling
└── linker.ld           # Linker script (if bare metal)
```

**Anti-patterns:**
- Business logic buried in API handlers
- Model definitions scattered across files
- Utility functions with hidden state
- "Misc" folders or generic `helpers.py`

### README Quality Rules

- **Every code example must work** — copy-paste from README, run, get expected output
- **No placeholder text** — `[TODO]`, `[insert here]`, `lorem ipsum` are rejected
- **Architecture section must match actual code** — if you rename a module, update the README
- **Performance numbers must be real** — run the benchmarks, paste the results
- **Quick Start must work on a clean machine** — test the steps mentally, don't assume pre-installed tools
- **Keep it concise** — comprehensive does not mean verbose; use tables and bullet lists

## CHANGELOG Contract

**Every project must include a CHANGELOG.md. Not just libraries — all project types.**

### Format: Keep a Changelog (https://keepachangelog.com)

```markdown
# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- Initial project implementation
- CLI interface with `process` and `validate` commands
- REST API with `/api/v1/resource` endpoints
- Unit and integration test suite (87% coverage)
- Performance benchmarks for core processing pipeline
- GitHub Actions CI workflow
- Structured logging with configurable levels

### Security
- Input validation on all API boundaries
- Rate limiting on public endpoints

## [0.1.0] - YYYY-MM-DD

### Added
- Initial release
```

### CHANGELOG Rules

- **Every code change gets a CHANGELOG entry** — no exceptions
- **Use correct categories:** Added, Changed, Deprecated, Removed, Fixed, Security
- **Write for humans** — "Added retry logic for API calls" not "Updated api.py"
- **Group related changes** — don't list every file, describe the feature/fix
- **Unreleased section always exists** — new changes go here until a version is cut
- **Link versions** — `[Unreleased]` and version numbers link to GitHub compare URLs when applicable
- **Date format:** ISO 8601 (YYYY-MM-DD)

### When to Update CHANGELOG

| Event | CHANGELOG Action |
|---|---|
| New feature added | `### Added` entry |
| Existing feature modified | `### Changed` entry |
| Feature deprecated | `### Deprecated` entry |
| Feature removed | `### Removed` entry |
| Bug fixed | `### Fixed` entry |
| Security fix | `### Security` entry |
| Version released | Move `[Unreleased]` to `[X.Y.Z] - YYYY-MM-DD` |

## Documentation Sync Contract

**README and CHANGELOG must always reflect the current state of the code. This is enforced, not suggested.**

### The Rule

> Every code change that affects behavior, APIs, configuration, or architecture **must** include corresponding updates to README.md and CHANGELOG.md in the same commit/generation.

### What Triggers a Sync

| Code Change | README Update | CHANGELOG Update |
|---|---|---|
| New function/endpoint/command | Add to Usage/API Reference | `### Added` |
| Changed function signature/behavior | Update Usage examples | `### Changed` |
| Removed function/endpoint | Remove from docs | `### Removed` |
| New config option / env var | Add to Configuration table | `### Added` |
| Removed config option | Remove from Configuration | `### Removed` |
| New dependency | Update Installation if needed | `### Added` (if user-facing) |
| Performance change | Update Performance numbers | `### Changed` |
| Bug fix | Update Troubleshooting if relevant | `### Fixed` |
| Architecture change | Update Architecture diagram | `### Changed` |
| Security fix | Update relevant sections | `### Security` |
| Setup process change | Update Quick Start / Installation | `### Changed` |

### Exemptions from Doc Sync

Not every change warrants a CHANGELOG entry or README update. The following are exempt:

- **Internal refactors** that do not change behavior, APIs, or configuration visible to users
- **Dependency bumps** that do not change behavior (security patches are exempt from CHANGELOG but should be noted in commit message)
- **Typo fixes** in code (not in docs — doc typo fixes are self-documenting)
- **Test-only changes** (adding tests, fixing flaky tests)
- **CI/build config changes** that do not affect the user experience

When in doubt, add the entry — it's cheap. But don't create noise entries like "Refactored internal variable naming."

### Enforcement

- **Code without matching doc updates is incomplete** — reject and re-request (unless exempt per above)
- **README examples that don't match code are bugs** — treat them with the same severity as failing tests
- **Stale CHANGELOG is a contract violation** — behavior-changing commits must have an entry
- **Agents must self-check:** Before marking work as complete, verify README and CHANGELOG are current

### For Coding Agents (Claude Code, Copilot, Cursor, etc.)

When modifying existing code in a project that uses this contract:

1. **Before making changes:** Read the current README.md and CHANGELOG.md
2. **After making changes:** Identify which sections of README need updating
3. **Update README.md:** Modify affected sections (examples, config tables, architecture, etc.)
4. **Update CHANGELOG.md:** Add entry under `[Unreleased]` with the correct category
5. **Verify consistency:** Ensure README examples still work with the changed code
6. **Include in same commit:** Doc updates are part of the code change, not a follow-up

**A code change without doc sync is not done. Period.**

## Security Contract

### Input Validation
- **All external inputs validated at the boundary** — API params, CLI args, file reads, env vars
- **Fail closed** — reject unknown fields, don't silently ignore
- **Type-safe parsing** — use language-native parsers, not regex for structured data
- **Length limits** — enforce max sizes on strings, arrays, file uploads

### Secrets Management
- **Zero secrets in code** — no API keys, passwords, tokens in source
- **`.env.example`** — document all required env vars with placeholder values
- **Secret loading** — use `os.getenv()`, `dotenv`, or config files, never hardcode
- **Gitignore `.env`** — prevent accidental commits

### Dependency Security
- **Minimize dependencies** — each dep must justify its inclusion
- **Pin versions** — exact or bounded ranges, no `*` or `latest`
- **Audit before ship** — run `pip audit` / `npm audit` / `cargo audit` / `go vuln check`
- **Lock files** — `poetry.lock`, `package-lock.json`, `Cargo.lock`, `go.sum` committed

### Network Security (for API/Network projects)
- **TLS by default** — no plaintext HTTP for production
- **CORS configured** — explicit origin allowlist, not `*`
- **Rate limiting** — protect against abuse
- **Auth before business logic** — validate tokens/keys first

### Memory Safety (for C/C++/unsafe Rust)
- **No `unsafe` without justification** — document why it's needed in a comment
- **AddressSanitizer in CI** — `-fsanitize=address` for C/C++
- **Valgrind clean** — zero leaks, zero invalid reads/writes
- **Bounds checking** — never index without checking length first

## Concurrency & Async Patterns

### General Rules
- **Document thread safety** — which types are `Send`/`Sync`, which are not thread-safe
- **No data races** — use appropriate synchronization primitives
- **Bounded queues/channels** — never unbounded queues in production
- **Graceful shutdown** — handle SIGTERM/SIGINT, drain in-flight work
- **Context propagation** — pass cancellation tokens/context through call chains

### Python
```python
import asyncio
from typing import AsyncIterator

async def process_stream(items: AsyncIterator[Item]) -> list[Result]:
    """Process items concurrently with bounded concurrency."""
    semaphore = asyncio.Semaphore(10)
    async def process_one(item: Item) -> Result:
        async with semaphore:
            return await do_work(item)
    return await asyncio.gather(*[process_one(i) async for i in items])
```

### Rust
```rust
use tokio::sync::Semaphore;
use std::sync::Arc;

async fn process_stream(items: Vec<Item>) -> Vec<Result> {
    let semaphore = Arc::new(Semaphore::new(10));
    let mut handles = Vec::new();
    for item in items {
        let permit = semaphore.clone().acquire_owned().await.unwrap();
        handles.push(tokio::spawn(async move {
            let _permit = permit;
            do_work(item).await
        }));
    }
    futures::future::join_all(handles).await
        .into_iter()
        .filter_map(|r| r.ok())
        .collect()
}
```

### Go
```go
func ProcessStream(ctx context.Context, items []Item) ([]Result, error) {
    sem := make(chan struct{}, 10) // bounded concurrency
    var wg sync.WaitGroup
    results := make([]Result, len(items))
    errs := make(chan error, len(items))

    for i, item := range items {
        wg.Add(1)
        go func(i int, item Item) {
            defer wg.Done()
            sem <- struct{}{}
            defer func() { <-sem }()

            result, err := doWork(ctx, item)
            if err != nil {
                errs <- err
                return
            }
            results[i] = result
        }(i, item)
    }
    wg.Wait()
    close(errs)
    // collect errors...
    return results, nil
}
```

### TypeScript
```typescript
async function processStream(items: Item[], concurrency = 10): Promise<Result[]> {
  const semaphore = new Semaphore(concurrency);
  return Promise.all(
    items.map(async (item) => {
      await semaphore.acquire();
      try {
        return await doWork(item);
      } finally {
        semaphore.release();
      }
    })
  );
}
```

## Code Taste (Subjective, Non-Negotiable)

### What "Good Taste" Means

**Clear naming:**
```python
# Bad
def process(data):
    x = data['x']
    y = x * 2
    return y

# Good
def calculate_double_value(config: dict[str, int]) -> int:
    """Double the configured value."""
    initial_value = config['initial_value']
    return initial_value * 2
```

**Minimal nesting (early return):**
```rust
// Bad (3+ levels deep)
fn process(user: Option<User>) -> bool {
    if let Some(u) = user {
        if let Some(email) = u.email {
            if email.contains('@') {
                return true;
            }
        }
    }
    false
}

// Good (guard clauses)
fn has_valid_email(user: &Option<User>) -> bool {
    let Some(u) = user else { return false };
    let Some(email) = &u.email else { return false };
    email.contains('@')
}
```

**One responsibility per function:**
```go
// Bad (does 3 things)
func processAndSaveAndNotify(userID string) error {
    user := fetchUser(userID)
    processUser(user)
    saveToDB(user)
    sendEmail(user)
    return nil
}

// Good (one per function)
func fetchAndValidate(ctx context.Context, userID string) (*User, error) {
    user, err := repo.GetUser(ctx, userID)
    if err != nil { return nil, fmt.Errorf("fetch: %w", err) }
    if err := validate(user); err != nil { return nil, fmt.Errorf("validate: %w", err) }
    return user, nil
}
```

**Comment the "why", not the "what":**
```typescript
// Bad (obvious)
const age = user.age; // Get the user's age
if (age > 18) { // Check if age is greater than 18
  sendEmail(); // Send an email
}

// Good (explains intent)
```
